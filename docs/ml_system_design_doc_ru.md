# ML System Design Doc - [RU]

## Дизайн ML систем - Productive Social MVP

*Шаблон ML System Design Doc от телеграм-канала [Reliable ML](https://t.me/reliable_ml)*

## 1. Зачем идем в разработку продукта?
 - бизнес-цель Productive Social - стать лучшим приложением для продуктивности.
 Мы стремимся не только помочь людям увеличить их продуктивность, но и создать систему,
способную изучать индивидуальную продуктивность, развивать понимание личных привычек 
(продуктивных или нет) и на основе этих данных предоставлять пользователю информационные
отчёты и рекомендации, как стать более продуктивным.

 - Productive Social лучше существующих приложений по трём ключевым аспектам:
   - Он будет охватывать все аспекты самопродуктивности;
   - Он предоставит систему машинного обучения, которая будет обучаться на конкретных данных каждого пользователя
(сбор данных через множество функций, которые включит Productive Social); 
   - Он будет иметь аспект социальной сети, разработанный для повышения вовлечённости и удержания пользователей.

## 2. Бизнес-требования и ограничения
 - Разработать универсальное приложение для любой деятельности, связанной с продуктивностью.
Приложение должно предоставлять разные функции как отдельные модули,
а также полностью интегрировать все функции вместе. Помимо социальной составляющей, 
все функции должны работать без подключения к интернету.
 - MVP должен включать:
   - Функцию управления задачами; 
   - Функцию ведения заметок; 
   - Функцию таймера по методу Помодоро; 
   - Функцию журнала (журнал событий) 
каждая из этих функций должна обеспечивать основные возможности аналогичных приложений.
В дополнение, социальный аспект должен включать, но не ограничиваться:
   - просмотр отчётов о продуктивности других пользователей и их прогрессе;
   - поддержку обмена медиафайлами (без кастомных настроек); 
   - личный профиль пользователя.
 - Ограничения:
   - Использование платных технологических платформ (например, AWS). 
   - Соблюдение требований GDPR и других нормативных актов о защите данных. 
   - Время реализации — не менее 6 месяцев.
 - Описание бизнес-процесса пилота, насколько это возможно - как именно мы будем использовать 
модель в существующем бизнес-процессе?
   - Productive Social будет доступно через Google Play Store для андройда и App Store для iOS.
 - Что считаем успешным пилотом? Критерии успеха и возможные пути развития проектa?
   - Трудно сказать что является успех для данного проекта, пока я уверен что смогу разработать приложение, но
 не могу сказать если систем ML будет работать как желается.
Если немного поговорить о функции 'Journaling', система машинного обучения должна уметь предоставлять 
дневной журнал с отчетами, сформированными на основе использования других функций,
которые предоставляет Productive Social. Это значит, что система должна точно указывать,
над чем работал пользователь в определённый день, сколько часов он на это потратил и сколько времени
потребовалось для достижения конкретной цели. Если пользователь не использовал приложение в течении два или более дней,
функция журнала всё равно должна (предсказывать) и предоставлять опциональную запись в журнал за эти дни. Если есть
информация из менеджера задач, заметок или предыдущих записей журнала о том, что пользователь, вероятно,
делал в эти два дня, то ML-система должна учитывать это и автоматически создавать соответствующую запись в журнале.
_Если приложение будет работать как ожидается — это и есть успех._
 
## 3. Что входит в скоуп проекта, что не входит
 - Входит:
   - Функция управления задачами; 
   - Функция ведения заметок; 
   - Функция таймера по методу Помодоро; 
   - Функция журнала; 
   - Функция социальной сети.
 - Не входит:
   - Полноценная социальная сеть. В рамках этого проекта социальные функции ограничиваются:
      - оценкой продуктивности других пользователей;
      - просмотром времени выполнения задач;
      - ограничивается возможности прямого обмена сообщениями или личного взаимодействия;
      - просмотром только основного ленты, без дополнительных лент.
 - На закрытие каких БТ подписываемся в данной итерации
   - На MVP, на мобильное приложение ожидается реализаций все выше перечислены фичи, 
   для Data Scientist ожидается реализации фичи дневника 
 - Описание результата с точки зрения качества кода и воспроизводимости решения Data Scientist
   - Результат должен соответствовать:
     - Высокому качеству кода (читаемость, чистота, документация)
     - Воспроизводимости (использование версионирования, контейнеризация, автоматизация)
 - Описание планируемого технического долга (что оставляем для дальнейшей продуктивизации) Data Scientist
   - В будущем планируется внедрение ИИ во всех фичи приложения.

## 4. Предпосылки решения
 - используемые блоки данных:
   - внутренние базы данных, датасеты о привычках и поведении; 
   - Данные о вовлечённости и использовании — публичные датасеты по аналитике мобильных приложений или синтетические данные;
   - Мотивационные и психологические данные — исследовательские датасеты или опросы по психологии и мотивации;
   - Внешние данные для контекста — публичные датасеты о погоде, API календарей;
   - В рамках использования Productive Social будет собираться максимально возможное количество данных (в рамках закона).
 - Гранулярность модели:
   - Хотя модели будут для общего назначения, при использовании модели они должны обучаться на данных конкретного юзера
